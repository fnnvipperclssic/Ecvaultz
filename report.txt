1. EXECUTIVE SUMMARY
================================================================================

Ecvaultz is a secure digital vault built on Laravel 11 + React/Inertia.js with
strong foundational security (AES-256-GCM, 2FA, RBAC, audit trail). To compete
with Proton Drive, Tresorit, NordLocker, and MEGA, it needs enhancements in
three areas: (1) fixing broken features, (2) adding table-stakes features,
and (3) introducing unique differentiators.

This report identifies 30+ improvements, categorized by priority and effort.

2. CURRENT STATE ASSESSMENT
================================================================================

STRENGTHS:
  - AES-256-GCM per-user encryption
  - TOTP 2FA with recovery codes
  - RBAC with Spatie Permission (28 permissions)
  - Internal + External file sharing with password/expiry
  - Soft-delete trash with 30-day recovery
  - Complete audit trail (ActivityLog)
  - OWASP Top 10 compliance headers (CSP, HSTS, X-Frame-Options)
  - ClamAV malware scanning
  - Command palette (Ctrl+K)
  - Dark theme with glassmorphism UI
  - Drag-and-drop upload with progress
  - File version history

GAPS (by severity):
  CRITICAL:
    - System Settings admin page does NOT persist changes (broken)
    - Mail notifications exist but are never sent (AccountLockoutMail,
      SecurityAlertMail, NotificationService methods)
    - Console cleanup commands exist but are NOT scheduled in Kernel
    - Avatar upload route exists but no UI component on Profile page
    - Rate limiting defined but NOT attached to upload routes

  HIGH:
    - No storage quota per user (unlimited storage for all)
    - No file search across entire vault (folder-scoped only)
    - No file favorites/stars or tags/labels
    - Password expiry middleware does not force logout
    - No per-file audit trail (who accessed file X, when)
    - Session not regenerated before 2FA challenge (session fixation risk)
    - No file/folder copy or duplicate functionality
    - No drag-to-move files between folders

  MEDIUM:
    - No PWA/offline support (service worker not configured)
    - No password strength meter on registration
    - No data room / workspace functionality
    - No encryption key recovery mechanism
    - No IP whitelist for admin access
    - No file expiration/self-destruct option
    - Security questions hardcoded in Indonesian only
    - No keyboard shortcuts documentation

  LOW:
    - Unused dependencies (axios, lodash, clsx, dayjs in package.json)
    - No onboarding tour for new users
    - No responsive table handling on mobile
    - No language/locale selection

3. CRITICAL BUG FIXES (Phase 0 — Immediate)
================================================================================

FIX-01: System Settings Persistence
  Problem: Admin > Settings page shows config values but saving does not write
           to database or .env. Controller only logs activity.
  Solution: Create system_settings database table, persist key-value pairs,
           invalidate config cache on save.
  Effort: Medium | Impact: Critical

FIX-02: Wire Mail Notifications
  Problem: AccountLockoutMail and SecurityAlertMail classes exist but are never
           instantiated or sent. NotificationService methods are defined but
           never called.
  Solution: Fire mailables from LoginRequest (lockout), ShareController (share
           created), AuthenticatedSessionController (new device login),
           ProfileController (password changed).
  Effort: Low | Impact: High

FIX-03: Schedule Console Commands
  Problem: CleanupExpiredFiles, CleanupExpiredShares, CleanupActivityLogs
           commands exist but are not registered in app/Console/Kernel.php.
  Solution: Register all three as daily scheduled commands in Kernel.
  Effort: Low | Impact: Medium

FIX-04: Avatar Upload UI
  Problem: ProfileController::updateAvatar and route exist but Profile.jsx
           page has no upload UI component.
  Solution: Add avatar upload section to Profile.jsx with drag-drop, preview,
           and cropping.
  Effort: Low | Impact: Medium

FIX-05: Rate Limiting on Upload Routes
  Problem: 'upload' rate limiter defined in RouteServiceProvider but never
           attached to file upload routes.
  Solution: Add throttle:upload middleware to file upload routes.
  Effort: Low | Impact: High

FIX-06: 2FA Session Regeneration
  Problem: Session not regenerated before redirecting to 2FA challenge page,
           risking session fixation attacks.
  Solution: Call $request->session()->regenerate() in the login handler BEFORE
           redirecting to 2FA challenge.
  Effort: Low | Impact: High

FIX-07: Password Expiry Hard Enforcement
  Problem: PasswordExpiry middleware only redirects to profile page but does
           not prevent navigation to other routes.
  Solution: Block all routes except password change and logout when password
           is expired. Return 403 or force-redirect for all other requests.
  Effort: Low | Impact: Medium

4. HIGH-PRIORITY COMPETITIVE FEATURES (Phase 1)
================================================================================

FEAT-01: File Favorites / Star System
  What: Users can star/favorite files. A dedicated "Favorites" filter in the
        file list shows only starred files. Star is toggleable inline.
  Why: Table-stakes feature for file organization. All major competitors
       (Google Drive, Dropbox, OneDrive) have it.
  Effort: Low | Impact: High
  Implementation: Add 'is_favorited' boolean column to files table. Add
  star toggle endpoint. Add star icon to file rows. Add favorites filter.

FEAT-02: Storage Quota Per User with Dashboard Visualization
  What: Each user has a configurable storage quota (default 5GB). Dashboard
        shows usage bar with percentage. Upload is blocked when quota exceeds.
        Admin can set per-user quotas from user management.
  Why: Essential for multi-tenant deployment. Prevents storage abuse.
  Effort: Medium | Impact: High
  Implementation: Add 'storage_quota' column to users table (bigint, bytes).
  Add quota check in FileController@store before upload. Add usage bar
  component to Dashboard. Add quota field to Admin user edit form.

FEAT-03: Per-File Audit Trail
  What: Each file has its own activity timeline showing: who uploaded, who
        downloaded, who previewed, who shared, when. Displayed on file
        details panel and accessible from the file action menu.
  Why: Compliance requirement (SOC 2, GDPR). Competitors like Tresorit and
       Kiteworks offer full chain-of-custody per document.
  Effort: Medium | Impact: High
  Implementation: Extend existing ActivityLog with file_uuid foreign key.
  Add "File Activity" tab/section. Filter logs by file_uuid.

FEAT-04: Global File Search
  What: Search bar in header searches across ALL files in the user's vault
        (not just current folder). Results shown in dropdown with file type
        icons, folder path, and quick actions.
  Why: Current search is folder-scoped only, making it impossible to find
        files without navigating to the correct folder first.
  Effort: Medium | Impact: High
  Implementation: Add global search endpoint. Create SearchResults dropdown
  component. Index file names for efficient LIKE/fulltext queries.

FEAT-05: Password Strength Meter
  What: Real-time password strength indicator on registration, password change,
        and reset forms. Shows: length check, uppercase, lowercase, numbers,
        symbols. Color-coded bar (red -> orange -> green).
  Why: Improves security UX. Users create stronger passwords when given
       visual feedback. NIST SP 800-63B recommends this.
  Effort: Low | Impact: Medium
  Implementation: Create PasswordStrengthMeter.jsx component. Add to
  Register.jsx, Profile.jsx, ResetPassword.jsx, NewPasswordController forms.

FEAT-06: File Tags / Labels
  What: Users can add color-coded tags to files (e.g., "Important", "Draft",
        "Final", "Personal", "Work"). Tags are filterable in the file list.
        Custom tags can be created.
  Why: Flexible organization beyond folders. Proton Drive and MEGA both
       support labels/tags for classification.
  Effort: Medium | Impact: Medium
  Implementation: Create tags table (id, user_id, name, color). Create
  file_tag pivot table. Add tag selector component. Add tag filter to
  file list. Add tag CRUD endpoints.

FEAT-07: File Copy / Duplicate
  What: Users can copy/duplicate files to the same or different folder.
        Copied files get "Copy of " prefix and new UUID. Encryption is
        preserved.
  Why: Basic file management operation. Currently only move and delete exist.
  Effort: Low | Impact: Medium
  Implementation: Add duplicate endpoint. In FileService, create new file
  record with same encryption key, copy physical file.

FEAT-08: Drag-to-Move Files Between Folders
  What: Users can drag a file row from the file list and drop it onto a
        folder in the sidebar to move it. Visual feedback during drag.
  Why: Modern file managers (Google Drive, Dropbox) support this. Reduces
       clicks for common operation.
  Effort: Medium | Impact: Medium
  Implementation: Add HTML5 drag events to file rows. Add drop zones to
  folder sidebar items. Wire to existing file move endpoint.

FEAT-09: Trash Enhancements
  What: Add "Empty Trash" button (permanently delete all trashed files at
        once with password confirmation). Add search within trash. Add sort
        by deletion date. Show remaining days before auto-purge per file.
  Why: Trash management is currently bare-bones (list only). Power users
       need batch operations.
  Effort: Low | Impact: Medium
  Implementation: Add empty-trash endpoint. Add search/filter to Trash page.
  Add empty-trash button with confirmation modal.

FEAT-10: File Expiration / Self-Destruct
  What: Users can set an expiry date on files. Expired files are automatically
        moved to trash (soft-delete) or permanently deleted based on setting.
        Visual countdown shown on expiring files.
  Why: Privacy feature. Snapchat-style ephemeral files are increasingly
       popular for sensitive document sharing.
  Effort: Medium | Impact: Medium
  Implementation: Add 'expires_at' column to files table. Add expiry picker
  to upload form and file properties. Wire to existing CleanupExpiredFiles
  command. Show countdown badge on file rows.

5. DIFFERENTIATING FEATURES (Phase 2)
================================================================================

FEAT-11: Dynamic Watermarking
  What: When previewing shared documents (PDF, images), overlay a semi-
        transparent watermark with: viewer email, timestamp, IP address,
        and "CONFIDENTIAL" stamp. Applied server-side on preview.
  Why: Deters unauthorized screenshots and redistribution. Tresorit and
       689Cloud use this as a premium differentiator.
  Effort: High | Impact: High
  Implementation: Use Intervention/Image for image watermarking. Use
  FPDI/FPDF or TCPDF for PDF watermarking. Apply watermark in preview
  controller based on share context.

FEAT-12: Virtual Data Room (VDR)
  What: Users can create "data rooms" — curated, branded spaces for external
        collaboration. Each room has: custom branding (logo, colors), file
        collections, granular access controls (view-only, download, watermark),
        full audit trail, expiry date. Share links are room-scoped.
  Why: Tresorit Engage launched VDR in 2025. Growing market for M&A, legal
       discovery, audits. Premium feature that commands higher pricing.
  Effort: High | Impact: Very High
  Implementation: Create data_rooms table (uuid, user_id, name, description,
  branding JSON, expires_at). Create data_room_files pivot. Create
  DataRoomController with views. Extend sharing to room-scoped links.

FEAT-13: Encryption Key Recovery Kit
  What: During account setup, users are shown a 12-word recovery phrase (BIP39
        compatible) derived from their encryption key. Users MUST download/save
        this. If they lose their password, the recovery phrase restores access
        to encrypted files. Clear warnings: "If you lose this, your files are
        permanently unrecoverable."
  Why: Zero-knowledge UX critical path. Without this, losing the encryption
       key means permanent data loss. Proton Drive, NordLocker all use this.
  Effort: Medium | Impact: High
  Implementation: Use BIP39 wordlist to encode encryption key as mnemonic.
  Create RecoveryKit.jsx page shown after registration and in Profile.
  Generate downloadable PDF with seed words.

FEAT-14: PWA / Offline Access
  What: Configure full Progressive Web App with service worker. Cache the app
        shell for offline launch. Cache recently accessed file metadata in
        IndexedDB. Show offline indicator. Queue actions (rename, delete) for
        sync when back online.
  Why: Native app-like experience. Competes with desktop/mobile vault apps.
       TrustVault-PWA has shown this is viable for secure apps.
  Effort: Medium | Impact: High
  Implementation: Create service-worker.js with Workbox. Update manifest.json
  with PWA icons. Add offline detection in React (navigator.onLine). Add
  IndexedDB cache layer for file metadata.

FEAT-15: Breach Detection (HIBP Integration)
  What: On login and password change, check the user's email against the
        Have I Been Pwned API (k-anonymity model — only sends first 5 chars
        of SHA-1 hash). If found in a breach, show alert with breach details
        and recommend password change.
  Why: Security differentiator. Dashlane, 1Password, Firefox Monitor use
       this. Low implementation effort for high security perception.
  Effort: Low | Impact: Medium
  Implementation: Create HIBPService that queries
  https://api.pwnedpasswords.com/range/{hash_prefix}. Check on login and
  password change. Show notification if breached.

FEAT-16: AI-Powered File Organization (LLM-based)
  What: On-device/local processing that analyzes uploaded files and suggests:
        - Auto-tags based on content analysis
        - Duplicate file detection (by checksum + filename similarity)
        - Content classification ("Invoice", "Contract", "Resume", "Photo")
        - Smart folder suggestions
        All processing via the browser (no data sent to external AI APIs).
  Why: Iberbox launched "private AI" in 2026. Edge AI for privacy is trending.
       Unique differentiator with strong privacy narrative.
  Effort: Very High | Impact: High
  Implementation: Use Transformers.js for on-device text extraction from
  documents. Implement checksum-based duplicate detection. Classification
  via filename heuristics + content patterns.

FEAT-17: Email Notification System
  What: Send transactional emails for:
        - Account lockout warning ("Your account has been locked")
        - New device/location login ("New login from [IP] in [City]")
        - File shared with you ("[User] shared [filename] with you")
        - Share link accessed ("Your shared file was downloaded")
        - Password changed confirmation
        - 2FA enabled/disabled alert
  Why: The mail classes already exist and are not wired. Completing this
       feature closes a major security UX gap.
  Effort: Low | Impact: High
  Implementation: Wire existing AccountLockoutMail in LoginRequest. Wire
  SecurityAlertMail in AuthenticatedSessionController. Create FileSharedMail
  and wire in ShareController. Create ShareAccessedMail.

6. FUTURE ROADMAP (Phase 3)
================================================================================

FEAT-18: Client-Side Zero-Knowledge Encryption
  Move encryption from server-side to browser using Web Crypto API (SubtleCrypto).
  Files encrypted before upload, server never sees plaintext. Requires major
  architectural change but is the #1 competitive differentiator.

FEAT-19: Post-Quantum Cryptography Readiness
  Implement crypto-agile key wrapping supporting NIST PQC standards
  (CRYSTALS-Kyber for KEM, CRYSTALS-Dilithium for signatures).

FEAT-20: Decentralized Storage Backend
  Distribute encrypted fragments across S3, IPFS, Arweave for vendor
  independence and censorship resistance.

FEAT-21: E-Signature Workflows
  Built-in document signing with serial/parallel approval chains.

FEAT-22: Secure Team Workspaces
  Group-level vaults with shared encryption keys, team roles, and
  collaborative folders.

FEAT-23: API & Webhooks
  REST API with Sanctum tokens for third-party integrations.
  Webhooks for file events (uploaded, shared, downloaded).

FEAT-24: SSO / OAuth Integration
  Support for SAML, OIDC, Google Workspace, Microsoft Entra ID login.
